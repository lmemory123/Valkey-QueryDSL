# 快速开始

这篇文档的目标是让你在一个新的 Spring Boot 项目中，把 `Valkey QueryDSL` 接入并跑通第一条查询。

## 1. 前置条件

- Java 17+
- Maven 3.9+
- 可用的 Valkey 服务
- 服务端具备 Search / JSON 能力

## 2. 选择接入方式

推荐先区分两种场景：

- 已发布稳定版：直接使用 Maven Central 上的 `io.github.lmemory123:1.0.0`
- 已发布候选版：直接使用 Maven Central 上的 `io.github.lmemory123:1.1.0-RC3`
- 当前源码快照：在本仓库本地执行 `mvn clean install` 后，使用 `com.momao:1.1.0-SNAPSHOT`

### 2.1 使用已发布稳定版

业务项目至少需要引入 Starter：

```xml
<dependency>
    <groupId>io.github.lmemory123</groupId>
    <artifactId>valkey-query-spring-boot-starter</artifactId>
    <version>1.0.0</version>
</dependency>
```

并为编译器配置 APT：

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <annotationProcessorPaths>
            <path>
                <groupId>io.github.lmemory123</groupId>
                <artifactId>valkey-query-processor</artifactId>
                <version>1.0.0</version>
            </path>
        </annotationProcessorPaths>
    </configuration>
</plugin>
```

### 2.2 使用当前源码快照

如果你要在其他项目中引用当前工作区里的最新源码，先在本仓库根目录执行：

```bash
mvn clean install
```

当前源码快照会安装到本机 Maven 仓库：

- `com.momao:valkey-query-annotations:1.1.0-SNAPSHOT`
- `com.momao:valkey-query-core:1.1.0-SNAPSHOT`

- `com.momao:valkey-query-spring-boot-starter:1.1.0-SNAPSHOT`

对应依赖写法：

```xml
<dependency>

```

APT 配置：


## 4. 配置连接

```yaml
valkey:
  query:
    enabled: true
    mode: standalone
    read-preference: primary
    observability:
      enabled: true
      slow-log-enabled: true
      async-slow-log-enabled: true
      slow-query-threshold-ms: 200
      trace-query-text-enabled: false
      slow-log-thread-name: valkey-query-slowlog
    use-tls: false
    request-timeout: 5000
    connection-timeout: 5000
    username: your-username
    password: your-password
    nodes:
      - host: 127.0.0.1
        port: 6379
```

如果你需要读写分离，可以切到：

```yaml
valkey:
  query:
    mode: read_write_split
    read-preference: replica_preferred
    read-write:
      write:
        nodes:
          - host: 10.0.0.10
            port: 6379
      read:
        nodes:
          - host: 10.0.0.11
            port: 6380
```

如果你要为未来集群模式预留入口，可以使用：

```yaml
valkey:
  query:
    mode: cluster
    cluster:
      nodes:
        - host: 10.0.1.10
          port: 6379
        - host: 10.0.1.11
          port: 6379
```

兼容性说明：

- 旧的 `valkey.query.nodes` 仍然可以继续使用
- `mode=standalone` 时，未显式配置 `standalone.nodes` 会自动回退到旧 `nodes`
- `mode=read_write_split` 时，未配置 `read-write.read.nodes` 会自动回退到写节点
- `read-preference=primary` 时，所有读请求固定走写节点
- `read-preference=primary_preferred` 时，优先走写节点，写节点读失败后回退到读节点
- `read-preference=replica_preferred` 时，优先走读节点，读节点失败后回退到写节点
- `read_write_split` 模式下，框架会把写节点和读节点一起交给 Glide 做节点发现，再通过 `readFrom` 把只读请求优先路由到副本
- `observability.enabled=true` 是默认值，但只有 `ObservationRegistry` 实际生效时才进入 Observation 分支
- `observability.slow-log-enabled=true` 时，慢查询会按阈值输出慢日志
- `observability.async-slow-log-enabled=true` 时，慢日志会通过单线程后台 recorder 异步落地，不阻塞命令线程
- `observability.slow-query-threshold-ms` 控制慢查询阈值，默认 `200ms`
- `observability.trace-query-text-enabled=false` 是默认值，避免把高基数 query 文本默认灌进 tracing 系统
- 异步慢日志队列满时会直接丢弃记录，并把 `valkey.query.slowlog.dropped` 计数器加一
- `mode=cluster` 时，索引命令会广播到各分片，`FT.SEARCH` / `count()` 会做跨分片聚合

如果应用上下文里已经接了 Micrometer Observation / Tracing 处理链，框架会复用同一个 `ObservationRegistry`：

- 指标和 trace 都走统一的 `valkey.command` Observation 名称
- 是否真正产出 metrics / tracing，由业务应用自己的 Spring Boot / Micrometer 配置决定
- 框架不再手写两套 Metrics / Tracing recorder
- Observation 默认携带 `db.system=valkey`、`db.operation`、`db.namespace`、`valkey.route.type`
- 失败场景下还会补充 `valkey.error.code`、`valkey.error.category`
- 额外失败指标会输出到 `valkey.query.command.failed`
- 慢查询计数会输出到 `valkey.query.command.slow.total`
- 索引治理相关指标会输出到 `valkey.query.index.diff`、`valkey.query.index.validation.failed`、`valkey.query.index.recreate`

推荐策略：

- 本地排障：开启 `observability.enabled=true`
- 压测或极致性能场景：保留 `trace-query-text-enabled=false`
- 生产环境若必须开启慢日志，优先保留 `async-slow-log-enabled=true`

### 本地主从验证

如果你要在本地快速验证读写分离，可以直接启动一主一从：

当前项目的真实测试基线依赖带 `TEXT` 索引能力的自构建 bundle 镜像，示例默认使用：

- `ghcr.io/lmemory123/valkey-bundle:9.1.0`

如果未来官方公开了等价镜像，可以再切回官方地址。

```bash
docker run -d --name valkey-rw-primary \
  -p 6381:6379 \
  ghcr.io/lmemory123/valkey-bundle:9.1.0 \
  valkey-server --port 6379 --appendonly no

docker run -d --name valkey-rw-replica \
  -p 6382:6379 \
  ghcr.io/lmemory123/valkey-bundle:9.1.0 \
  valkey-server --port 6379 --replicaof host.docker.internal 6381 --appendonly no
```

然后执行主从集成测试：

```bash
VALKEY_RW_INTEGRATION=true \
mvn -B -ntp -pl valkey-query-test-example -am \
  -Dtest=ReadWriteSplitIntegrationTests \
  -Dsurefire.failIfNoSpecifiedTests=false test
```

默认测试地址：

- 写节点：`localhost:6381`
- 读节点：`localhost:6382`

如需覆盖默认地址，可设置：

- `VALKEY_RW_WRITE_HOST`
- `VALKEY_RW_WRITE_PORT`
- `VALKEY_RW_READ_HOST`
- `VALKEY_RW_READ_PORT`

## 5. 定义实体

```java
@ValkeyDocument(
    indexName = "idx:sku",
    prefixes = {"sku:"},
    storageType = StorageType.JSON
)
public class Sku {

    @ValkeyId
    private String id;

    @ValkeySearchable(weight = 2.5d, noStem = true)
    private String title;

    @ValkeyIndexed(sortable = true)
    private Integer price;

    @ValkeyIndexed
    private List<String> tags;

    @ValkeyIndexed
    private Merchant merchant;
}
```

```java
public class Merchant {
    private String name;
    private String level;
}
```

编译后会生成 `SkuQuery`。

## 6. 定义 Repository

```java
@Repository
public class SkuRepository extends BaseValkeyRepository<Sku> {

    public SkuRepository(ValkeyClientRouting clientRouting, ObjectMapper objectMapper) {
        super(SkuQuery.METADATA, clientRouting, Sku.class, objectMapper);
    }
}
```

## 7. 发起查询

```java
SkuQuery q = new SkuQuery();

Page<Sku> page = skuRepository.queryChain()
    .where(q.title.contains("iphone"))
    .and(q.merchant.name.eq("Apple Flagship"))
    .orderByAsc("price")
    .page(0, 20);
```

或者直接使用 Repository：

```java
SearchCondition condition = q.title.contains("iphone").sortBy("price", true);

List<Sku> list = skuRepository.list(condition);
Page<Sku> result = skuRepository.page(condition, 0, 20);
Sku one = skuRepository.one(q.id.eq("sku-1"));
long total = skuRepository.count(q.tags.eq("HOT"));
```
