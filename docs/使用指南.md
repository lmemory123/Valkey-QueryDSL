# 使用指南

这篇文档聚焦于真实使用时最关心的三件事：怎么建模、怎么查询、运行时到底做了什么。

## 1. 注解模型

### `@ValkeyDocument`

放在实体类上，定义索引信息。

常用属性：

- `indexName`
- `prefixes`
- `storageType`
- `querySuffix`

默认值：

- `storageType = JSON`
- `querySuffix = Query`

### `@ValkeyId`

标记主键字段。查询结果在映射回实体时，会尝试根据文档 key 自动回填 `id`。

### `@ValkeySearchable`

适合全文检索字段。

常用属性：

- `value`
- `weight`
- `noStem`
- `sortable`

### `@ValkeyIndexed`

适合结构化索引字段。

常用属性：

- `value`
- `sortable`

## 2. 生成的 Query 类

APT 会根据实体生成 `XXXQuery`，例如：

- `SkuQuery`
- `StudentQuery`

它的作用是避免你在业务层直接手写查询字符串。

```java
SkuQuery q = new SkuQuery();

SearchCondition condition = q.title.contains("iphone")
    .and(q.price.between(1000, 5000))
    .and(q.merchant.name.eq("Apple Flagship"));
```

## 3. Repository API

`ValkeyRepository<T>` 当前核心接口：

- `save(String id, T entity)`
- `search(SearchCondition condition)`
- `list(SearchCondition condition)`
- `page(SearchCondition condition, int offset, int limit)`
- `one(SearchCondition condition)`
- `count(SearchCondition condition)`
- `checkAndCreateIndex()`
- `queryChain()`

推荐使用方式：

- 平时用 `list / page / one / count`
- 需要拿到完整命中数和记录列表时再用 `search`

## 4. 链式查询

```java
SkuQuery q = new SkuQuery();

Page<Sku> page = skuRepository.queryChain()
    .where(q.tags.eq("HOT"))
    .and(q.merchant.name.eq("Merchant-A"))
    .orderByAsc("price")
    .page(0, 10);
```

链式 API 最终会收敛为一个 `SearchCondition`，再交给 Repository 执行。

## 5. JSON 与 HASH

### JSON

适合：

- 集合字段
- 嵌套对象
- 复杂业务实体

比如：

- `List<String> tags`
- `Merchant merchant`

### HASH

适合：

- 扁平结构
- 字段简单的业务对象

如果实体里有明显的嵌套对象，优先选择 `JSON`。

## 6. 集合字段与嵌套对象

当前项目已经验证过：

1. `List<String>` 集合字段
2. `merchant.name` 这类嵌套对象字段

所以你可以写：

```java
SkuQuery q = new SkuQuery();

List<Sku> records = skuRepository.queryChain()
    .where(q.tags.eq("HOT"))
    .and(q.tags.eq("NEW"))
    .list();
```

也可以写：

```java
List<Sku> merchantA = skuRepository.queryChain()
    .where(q.merchant.name.eq("Merchant-A"))
    .list();
```

## 7. 运行时行为

### 索引创建

启用 `@EnableValkeyQuery` 后，Starter 会在应用就绪时：

1. 扫描 `@ValkeyDocument`
2. 加载生成的 `XXXQuery.METADATA`
3. 调用 `checkAndCreateIndex()`

### 查询执行

当前策略是：

1. 无排序查询走 Glide SDK 原生 `FT.search`
2. 有排序查询走真实 `FT.SEARCH ... SORTBY ... LIMIT ...`
3. `count()` 走 `FT.SEARCH ... LIMIT 0 0`

这是当前项目为了兼顾 SDK 能力和服务端真实排序行为做的收口。

## 8. 推荐实践

- 复杂实体优先使用 `StorageType.JSON`
- 所有实体显式声明 `indexName` 和 `prefixes`
- 在业务项目里保留至少一个真实集成测试
- 业务层优先用 `queryChain()`，不要直接手拼 `FT.SEARCH`
- 把 `valkey-query-test-example` 当成接入模板
