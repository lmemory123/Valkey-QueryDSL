# 使用指南

这篇文档聚焦于真实使用时最关心的三件事：怎么建模、怎么查询、运行时到底做了什么。

## 1. 注解模型

### `@ValkeyDocument`

放在实体类上，定义索引信息。

常用属性：

- `indexName`
- `prefixes`
- `storageType`
- `querySuffix`

默认值：

- `storageType = JSON`
- `querySuffix = Query`

### `@ValkeyId`

标记主键字段。查询结果在映射回实体时，会尝试根据文档 key 自动回填 `id`。

### `@ValkeySearchable`

适合全文检索字段。

常用属性：

- `value`
- `weight`
- `noStem`
- `sortable`

说明：

- `sortable` 当前只对 `TEXT` / `NUMERIC` 有效
- `TAG` 字段不会生成 `SORTABLE`

### `@ValkeyIndexed`

适合结构化索引字段。

常用属性：

- `value`
- `sortable`

说明：

- `String` 这类会推导成 `TAG` 的字段，当前会忽略 `sortable=true`
- 数值字段可以正常使用 `sortable=true`

## 2. 生成的 Query 类

APT 会根据实体生成 `XXXQuery`，例如：

- `SkuQuery`
- `StudentQuery`

它的作用是避免你在业务层直接手写查询字符串。

```java
SkuQuery q = new SkuQuery();

SearchCondition condition = q.title.contains("iphone")
    .and(q.price.between(1000, 5000))
    .and(q.merchant.name.eq("Apple Flagship"));
```

## 3. Repository API

`ValkeyRepository<T>` 当前核心接口：

- `save(String id, T entity)`
- `updateById(Object id, List<UpdateOperation> operations)`
- `search(SearchCondition condition)`
- `list(SearchCondition condition)`
- `page(SearchCondition condition, int offset, int limit)`
- `one(SearchCondition condition)`
- `count(SearchCondition condition)`
- `checkAndCreateIndex()`
- `queryChain()`
- `updateChain()`

推荐使用方式：

- 平时用 `list / page / one / count`
- 需要局部修改少数字段时，用 `updateChain()`
- 需要拿到完整命中数和记录列表时再用 `search`

## 4. 链式查询

```java
SkuQuery q = new SkuQuery();

Page<Sku> page = skuRepository.queryChain()
    .where(q.tags.eq("HOT"))
    .and(q.merchant.name.eq("Merchant-A"))
    .orderByAsc("price")
    .page(0, 10);
```

链式 API 最终会收敛为一个 `SearchCondition`，再交给 Repository 执行。

## 5. 局部更新

当前版本已经支持最小可用的 `UpdateChain / Partial Update`：

```java
SkuQuery q = new SkuQuery();

long updated = skuRepository.updateChain()
    .where(q.id.eq("sku-1"))
    .and(q.price.eq(1999))
    .set(q.price, 2999)
    .set(q.merchant.name, "Apple Flagship")
    .execute();
```

当前语义：

- 推荐主入口：`where / and / or / andIf / orIf`
- `where(...)` 中需要能唯一定位到单个文档，最常见写法是 `where(q.id.eq(...))`
- 至少要有一个 `set(...)`
- 支持 `setIf(...)`
- `expect(...)` 仍然保留，但现在更适合作为快捷别名，而不是主入口
- `expectVersion(...)` 是 `expect(...)` 的语义化别名，可直接拿来做乐观锁入口
- 数值字段支持 `increment(...)` / `decrement(...)`
- 数值字段支持 `incrementIf(...)` / `decrementIf(...)`
- 使用 `whereId(...)` 快捷方式时，更新前会先检查 key 是否存在，不会偷偷创建半残缺文档
- key 不存在时返回 `0`
- key 存在且更新成功时返回 `1`
- 条件不匹配时也返回 `0`

底层映射：

- JSON：按字段逐个下发 `JSON.SET`
- HASH：合并为一次 `HSET`

当前已经验证的更新场景：

- `NumericFieldBuilder`
- `TextFieldBuilder`
- `TagFieldBuilder`
- `where / and / or / andIf / orIf`
- `setIf(...)`
- `expect(...)` 快捷条件更新
- `expectVersion(...)` 乐观锁入口
- `advanceVersion(...)` 版本字段 compare-and-set + 自动加一
- `increment(...)` / `decrement(...)` 原子增减
- JSON 嵌套路径，如 `q.merchant.name`
- JSON 集合 TAG，如 `q.tags`，会自动归一化成索引用的分隔字符串

当前边界：

- 当前更新链仍然只支持可结构化解析的精确匹配条件
- 典型支持写法是 `q.id.eq(...)`、`q.price.eq(...)`、`q.status.eq(...)`
- `expectVersion(...)` 和 `advanceVersion(...)` 只接受 `version / *.version / *_version` 这类版本字段
- `contains / gt / between / matches / not` 这类条件暂不能直接用于更新链
- 暂不支持 `null` 更新
- 暂不支持批量 update
- 条件更新当前只支持单 key compare-and-set，不支持跨文档条件批量更新

原子增减示例：

```java
StudentQuery q = new StudentQuery();

studentRepository.updateChain()
    .where(q.id.eq("1"))
    .increment(q.age, 1)
    .execute();
```

当前实现约束：

- `increment(...)` / `decrement(...)` 只接受 `NumericFieldBuilder`
- 单字段无条件增减优先走原生命令：
  - HASH：`HINCRBY`
  - JSON：`JSON.NUMINCRBY`
- 一旦出现 `where / and / or / expectVersion / set + increment` 这类混合场景，会统一走单 key Lua 脚本
- Lua 执行走脚本缓存，运行时优先 `EVALSHA`，命中 `NOSCRIPT / NoScriptError` 后自动 `SCRIPT LOAD` 并重试
- 对外返回语义保持不变：成功返回 `1`，条件不满足或 key 不存在返回 `0`

## 6. JSON 与 HASH

### JSON

适合：

- 集合字段
- 嵌套对象
- 复杂业务实体

比如：

- `List<String> tags`
- `Merchant merchant`

### HASH

适合：

- 扁平结构
- 字段简单的业务对象

如果实体里有明显的嵌套对象，优先选择 `JSON`。

## 7. 集合字段与嵌套对象

当前项目已经验证过：

1. `List<String>` 集合字段
2. `merchant.name` 这类嵌套对象字段

所以你可以写：

```java
SkuQuery q = new SkuQuery();

List<Sku> records = skuRepository.queryChain()
    .where(q.tags.eq("HOT"))
    .and(q.tags.eq("NEW"))
    .list();
```

也可以写：

```java
List<Sku> merchantA = skuRepository.queryChain()
    .where(q.merchant.name.eq("Merchant-A"))
    .list();
```

## 8. 运行时行为

### 索引创建

启用 `@EnableValkeyQuery` 后，Starter 会在应用就绪时：

1. 扫描 `@ValkeyDocument`
2. 加载生成的 `XXXQuery.METADATA`
3. 按 `index-management.mode` 执行索引治理

索引相关操作固定走写客户端，不参与读路由。

当前索引治理模式：

- `NONE`
  什么都不做，不检查，不创建
- `VALIDATE`
  只检查，不修改；索引缺失或 schema 不一致时直接阻止启动
- `RECREATE`
  删除后重建，仅适合本地开发和测试环境

配置示例：

```yaml
valkey:
  query:
    index-management:
      mode: validate
```

当前建议：

- 生产环境优先用 `NONE` 或 `VALIDATE`
- 本地开发和测试环境用 `RECREATE`
- 如果需要真正的平滑索引升级，建议通过运维脚本结合 `FT.ALIAS` 做外部蓝绿切换，框架端不提供危险的自动破坏性升级

### 查询执行

当前策略是：

1. 查询统一走真实 `FT.SEARCH` 命令路径
2. 排序查询直接下发 `FT.SEARCH ... SORTBY ... LIMIT ...`
3. `count()` 走 `FT.SEARCH ... LIMIT 0 0`
4. `cluster` 模式下会对分片结果做聚合，再回到 Repository 层完成排序分页合并
5. `UpdateChain` 的复杂写路径统一走单 key 原子执行，不依赖客户端多次往返拼接

这意味着当前框架已经不再依赖 Glide SDK 的 `FT.search` 结果对象，而是统一收口到真实命令执行。

### 拓扑与路由

从阶段一开始，运行时已经区分了三类客户端职责：

- 写请求：`save`
- 写请求：`save / updateById / updateChain`
- 索引请求：`checkAndCreateIndex`、索引重建
- 读请求：`search / list / page / one / count`

Starter 会根据 `valkey.query.mode` 装配路由：

- `standalone`
- `read_write_split`
- `cluster`

当前读路由策略由 `valkey.query.read-preference` 控制：

- `primary`
- `primary_preferred`
- `replica_preferred`

当前行为说明：

- `primary`：读请求固定走写客户端
- `primary_preferred`：优先走写客户端，失败后回退到读客户端
- `replica_preferred`：优先走读客户端，失败后回退到写客户端
- `read_write_split` 下的读客户端不会只连副本，而是合并主从地址后交给 Glide，通过 `readFrom=PREFER_REPLICA` 进行副本优先路由
- 索引命令始终走写侧，不参与读偏好切换
- `cluster` 模式下，`JSON.SET / HSET / JSON.GET` 通过 `MOVED` 重定向定位分片
- `cluster` 模式下，`FT.CREATE / FT.DROPINDEX / FT._LIST` 会广播到主分片
- `cluster` 模式下，`FT.SEARCH / count()` 会做跨分片聚合

真实验证结果：

- 已有单元测试覆盖 `PRIMARY / PRIMARY_PREFERRED / REPLICA_PREFERRED` 三种回退策略
- 已有真实主从集成测试验证 `ROLE` 命令的写主读从
- 已有真实主从集成测试验证 `save -> JSON.GET / FT.SEARCH` 这条 Repository 读写链路
- 已有真实 cluster 集成测试验证 `MOVED`、索引广播和跨分片检索

### 可观测性

从当前版本开始，可观测性不再包在 `ValkeyClientRouting` 外层，而是由 `BaseValkeyRepository` 和实体存储策略统一走 `ValkeyObservationInvoker`。

执行分为 3 级分支：

- 分支 A：`observability` 和 `slow-log` 都关闭，直接裸跑 callback
- 分支 B：只开启慢日志，只做 `System.nanoTime()` 计时，命中阈值后异步落地慢日志
- 分支 C：Observation 生效时，统一进入 `ObservationRegistry`，由 Micrometer 处理 metrics / tracing

当前配置项：

- `valkey.query.observability.enabled=true`
- `valkey.query.observability.slow-log-enabled=true`
- `valkey.query.observability.async-slow-log-enabled=true`
- `valkey.query.observability.slow-query-threshold-ms=200`
- `valkey.query.observability.trace-query-text-enabled=false`

关键约束：

- `querySupplier` 默认不会执行
- 只有 `trace-query-text-enabled=true` 或执行报错时，才会取具体 query 文本并写入 `db.statement`
- 慢日志默认走单线程异步 recorder，不阻塞命令执行线程
- Metrics / tracing 是否真正生效，由 Spring Boot 应用自身的 Micrometer Observation 配置决定
- 异步慢日志队列满时会丢弃记录，并通过 `valkey.query.slowlog.dropped` 暴露丢弃计数
- 当前 low-cardinality tags 还会补充 `valkey.route.type`

## 9. 推荐实践

- 复杂实体优先使用 `StorageType.JSON`
- 所有实体显式声明 `indexName` 和 `prefixes`
- 在业务项目里保留至少一个真实集成测试
- 业务层优先用 `queryChain()`，不要直接手拼 `FT.SEARCH`
- 修改少数字段时优先用 `updateChain()`，不要先整对象读出再全量 `save`
- 新项目优先使用 `mode + read-preference + read-write / cluster` 配置结构
- 把 `valkey-query-test-example` 当成接入模板

## 10. 异常模型

当前框架已经不再把核心运行时错误统一包装成 `IllegalStateException`，而是先分成 5 类：

- `ValkeyConfigurationException`
  典型场景：客户端未注入、索引元数据读取失败
- `ValkeyConnectionException`
  典型场景：`GlideClient` 创建失败、cluster 节点连接失败
- `ValkeyIndexException`
  典型场景：`FT.CREATE`、`FT.DROPINDEX`、索引重建失败
- `ValkeyQueryExecutionException`
  典型场景：`save`、`FT.SEARCH`、`count`、`JSON.GET` 执行失败
- `ValkeyResultMappingException`
  典型场景：JSON/HASH 反序列化失败、实体 ID 回填失败、排序字段反射读取失败

同时每个异常都会携带稳定错误码 `ValkeyErrorCode`，可以直接通过 `errorCode()`、`errorCategory()`、`errorCodeValue()` 读取。

当前错误码范围：

- `CONFIG_*`
- `CONNECTION_*`
- `INDEX_*`
- `QUERY_*`
- `MAPPING_*`

典型示例：

- `CONFIG_001`：客户端未注入
- `CONFIG_002`：无法加载实体索引元数据
- `CONNECTION_002`：`GlideClient` 创建失败
- `INDEX_002`：索引校验失败
- `INDEX_003`：索引创建失败
- `QUERY_004`：`FT.SEARCH` 执行失败
- `QUERY_006`：实体按 id 读取失败
- `MAPPING_002`：实体 ID 回填失败

这层拆分的意义是：

- 上层业务可以区分“连不上”还是“查失败了”
- 集成测试可以更准确地断言失败类型
- metrics / tracing 可以直接按错误码和类别打点

当前 Observation 在错误场景下会附带：

- `valkey.error.code`
- `valkey.error.category`
- `valkey.route.type`

这样在 Grafana / Tempo / Jaeger 一类观测系统里，可以直接按稳定语义聚合失败。

另外，Starter 当前已经补了两类 Micrometer counter：

- `valkey.query.command.failed`
  按 `db.operation`、`db.namespace`、`valkey.route.type`、`valkey.error.code`、`valkey.error.category` 聚合失败命令
- `valkey.query.command.slow.total`
  按 `db.operation`、`db.namespace`、`valkey.route.type` 聚合慢查询
- `valkey.query.index.diff`
  按治理模式和 diff 类型统计索引差异
- `valkey.query.index.validation.failed`
  统计索引校验失败
- `valkey.query.index.recreate`
  统计索引重建触发次数
- `valkey.query.update.partial.total`
  按 `updated / not_found / condition_not_matched` 聚合局部更新结果
- `valkey.query.update.partial.failed`
  当前局部更新指标还会按 `valkey.update.kind` 区分 `set / increment / decrement / mixed`
  按错误码聚合局部更新失败
