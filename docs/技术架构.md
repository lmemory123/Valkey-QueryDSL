# 技术架构

这篇文档说明 `Valkey QueryDSL` 的技术实现，不只回答“怎么用”，也回答“为什么它这样工作”。

## 1. 总体架构

项目分成两条主线：

1. 编译期链路
2. 运行时链路

### 编译期链路

`实体类 -> 注解处理器 -> 生成 XXXQuery / METADATA`

### 运行时链路

`业务代码 -> QueryChain / Repository -> Glide Adapter -> Valkey`

## 2. 编译期实现

编译期由 `valkey-query-processor` 完成。

核心类：

- `ValkeyQueryProcessor`

它会扫描：

- `@ValkeyDocument`
- 字段上的 `@ValkeyId`
- 字段上的 `@ValkeySearchable`
- 字段上的 `@ValkeyTag`
- 字段上的 `@ValkeyNumeric`
- 字段上的 `@ValkeyIndexed`

然后生成两类内容：

1. 字段查询构造器
2. `METADATA`

生成结果示意：

```java
public final class SkuQuery {
    public static final IndexSchema METADATA = ...;
    public final TextFieldBuilder title = new TextFieldBuilder("title");
    public final NumericFieldBuilder price = new NumericFieldBuilder("price");
    public final TagFieldBuilder tags = new TagFieldBuilder("tags");
}
```

底层实现依赖：

- `JavaPoet` 负责源码生成
- `AutoService` 负责注解处理器注册

## 3. 元数据模型

编译期生成的 `METADATA` 是运行时索引创建和查询执行的桥梁。

它至少包含：

- `indexName`
- `storageType`
- `prefixes`
- `fields`

运行时不会再重新推断实体结构，而是直接依赖这份编译期产物。

这意味着：

1. Java 实体和索引字段定义更容易保持一致
2. 运行时反射成本更低

## 4. 运行时接入

运行时入口来自 `valkey-query-spring-boot-starter`。

核心类：

- `EnableValkeyQuery`
- `ValkeyQueryScannerRegistrar`
- `ValkeyQueryAutoConfiguration`
- `ValkeyIndexAutoCreator`
- `ValkeyQueryProperties`

执行流程：

1. `@EnableValkeyQuery` 注册扫描基包
2. `ValkeyQueryAutoConfiguration` 根据 `mode` 创建 `GlideClient` 和 `ValkeyClientRouting`
3. Starter 绑定 `valkey.query.*` 配置
4. 应用启动完成后，`ValkeyIndexAutoCreator` 扫描 `@ValkeyDocument`
5. 读取生成的 `XXXQuery.METADATA`
6. 调用 `checkAndCreateIndex()` 自动建索引

当前拓扑模式：

- `standalone`
- `read_write_split`
- `cluster`

其中：

- `standalone` 使用单一 `GlideClient`
- `read_write_split` 使用写客户端 + 读路由客户端
- `cluster` 使用基于 `MOVED` 的节点发现和分片路由
- 可观测性不再通过路由装饰器实现，而是由 Repository 执行层统一调用 `ValkeyObservationInvoker`

## 5. Repository 执行层

核心类：

- `BaseValkeyRepository<T>`

它实现了 `ValkeyRepository<T>`，对外暴露：

- `save`
- `search`
- `list`
- `page`
- `one`
- `count`
- `updateChain`

这一层负责三件事：

1. 构建查询命令
2. 调用底层 Glide
3. 把响应映射回实体

## 6. JSON / HASH 存储策略

项目没有把 JSON 和 HASH 混在一个大类里硬写分支，而是拆成策略层。

核心接口与实现：

- `ValkeyEntityOperations<T>`
- `HashEntityOperations<T>`
- `JsonEntityOperations<T>`
- `ValkeyEntityOperationsFactory`

职责分工：

- 保存实体
- 局部更新与原子增减
- 构建 `FieldInfo`
- 从存储字段恢复对象

这样做的好处是：

1. `BaseValkeyRepository` 不会膨胀成上帝类
2. JSON / HASH 的差异被局部收口
3. 写路径可以按能力选择原生命令或脚本缓存路径

### 局部更新与原子写

当前写路径已经分成三类：

1. 纯 `set(...)`
2. 单字段原子 `increment / decrement`
3. 条件更新或混合更新

执行策略：

- HASH 纯 set：`HSET`
- JSON 纯 set：逐字段 `JSON.SET`
- HASH 单字段原子增减：`HINCRBY`
- JSON 单字段原子增减：`JSON.NUMINCRBY`
- 条件更新、版本 compare-and-set、`set + increment` 混合更新：统一走单 key Lua

为避免每次把 Lua 源码裸发到服务端，运行时新增了 `ValkeyScriptExecutor`：

- 优先本地计算 SHA1 后直接发 `EVALSHA`
- 只有命中 `NOSCRIPT / NoScriptError` 时才执行 `SCRIPT LOAD`
- 成功加载后回填 SHA，并再次 `EVALSHA`

这样可以把原子更新的脚本执行成本压到稳定可控范围内。

## 7. 结果映射

核心类：

- `ValkeyResultMapper<T>`

它负责：

1. 解析 `FT.SEARCH` 响应
2. 兼容 `Object[] / Map / List / byte[]`
3. 还原实体对象
4. 根据 `docKey` 回填实体 `id`

当前 `id` 回填策略：

1. 优先尝试 `setId(String)`
2. 不存在 setter 时，反射写入 `id` 字段

## 8. 查询执行策略

当前查询路径不是单一实现，而是按能力分流。

### 搜索查询

当前搜索统一走真实命令：

```text
FT.SEARCH idx query SORTBY field ASC LIMIT offset limit
```

### `count()`

为了避免返回文档内容，同时兼容当前服务端行为，走：

```text
FT.SEARCH idx query LIMIT 0 0
```

这个实现是有意为之，不是临时绕路。

### 读写分离

`read_write_split` 模式下，运行时不是简单地“写 client 连主、读 client 连从”。

真实主从验证表明：如果让 `GlideClient` 只连接副本节点，会在初始化阶段报 `No primary node found`。因此当前实现采用：

1. 写客户端只连接写节点，固定 `ReadFrom.PRIMARY`
2. 读客户端合并主节点和读节点地址
3. 当 `read-preference` 为副本优先相关策略时，读客户端使用 Glide 的 `ReadFrom.PREFER_REPLICA`
4. Repository 读请求通过 `ValkeyClientRouting` 选择执行侧

这个设计的结果是：

- 框架层仍然暴露“写主读从”的语义
- 底层客户端保持对主节点的感知能力
- 不需要让业务层知道 Glide 的副本连接限制

### 可观测性

当前可观测性改成了“Repository / EntityOps -> Invoker -> Routing”这条主链。

核心实现包括：

- `ValkeyObservationInvoker`
- `ValkeyCommandCallback`
- `SlowLogRecorder`
- `AsyncSlowLogRecorder`

当前设计原则：

- 通过方法入口的 boolean 标志位做 3 级分支，保证关闭时接近零开销
- 只在 Observation 分支里创建 `Observation`
- 高基数 query 文本默认不上报
- 默认 low-cardinality tags 对齐数据库语义：`db.system`、`db.operation`、`db.namespace`
- 运行时还会补充框架侧标签：`valkey.route.type`
- 只有失败命令或显式开启 `trace-query-text-enabled` 时，才把 query 文本写入 `db.statement`
- 慢日志默认异步，避免同步日志 I/O 抖动命令线程
- Metrics / tracing 统一复用 Spring Boot 的 `ObservationRegistry`，不再各写一套 recorder
- 异步慢日志队列满时，通过 `valkey.query.slowlog.dropped` 暴露背压丢弃计数

## 9. 索引一致性处理

`BaseValkeyRepository.checkAndCreateIndex()` 当前不只做“存在则跳过”。

它会先做两步检查：

1. 索引是否存在
2. 已存在索引是否和当前 `METADATA` 一致

如果不一致，会先删除旧索引，再按当前 schema 重建。

阶段二已经把这条链继续上提到了：

- `IndexDiff`
- `IndexMigrationPlan`
- `ValkeyIndexManager`
- `ValkeyManagedIndexPlan`

也就是说，启动期是否应用索引计划，已经不再是 Repository 层的隐式黑盒逻辑。

这样做是为了避免：

- 本地实体已经变更
- 远端索引还是旧结构
- 查询结果却悄悄跑偏

## 10. 当前设计取舍

项目现在的取舍很明确：

### 优先编译期确定性

- 用 APT 生成 Query 和元数据
- 尽量不让业务层拼字符串

### 优先运行时可验证

- 查询命令直接打印
- 集成测试覆盖真实 Valkey

### 优先索引可治理

- 当前不再只有“索引是否一致”这个二元结论
- 运行时已经能输出 `IndexDiff`
- 治理入口会继续收敛成 `IndexMigrationPlan`
- 当前计划动作先收口为 `NONE / CREATE / RECREATE`
- 已新增 `ValkeyIndexManager` 作为索引治理门面
- `IndexSchema` 已在元数据层区分 `indexName` 与 `aliasName`

### 优先异常可分型

- 配置、连接、索引、执行、映射五类异常已经拆开
- Repository 和 Starter 主链路不再统一抛 `IllegalStateException`
- 每个核心异常都已挂稳定 `ValkeyErrorCode`
- Observation 失败场景会带 `valkey.error.code`、`valkey.error.category`
- Starter 侧还补了 `valkey.query.command.failed`、`valkey.query.command.slow.total`、`valkey.query.index.*` 系列 counter
- 局部更新链路已经补了 `valkey.query.update.partial.total`、`valkey.query.update.partial.failed`，并带 `valkey.update.kind`
- 这样监控、日志聚合和错误分类时，不需要再从 message 里猜错误类型

### 优先可扩展的内部结构

- 存储策略拆分
- 结果映射拆分
- Starter 和 Adapter 分模块隔离

## 11. 已知边界

- APT 仍基于当前注解模型工作，不是通用 ORM
- 复杂查询能力仍受 Valkey Search 和 Glide 的边界约束
- 目前主要围绕 Spring Boot 和 Maven 工作流设计

但对于“Java + Spring Boot + Valkey Search”的目标场景，这套结构已经足够清晰且可维护。
