# 配置与治理

## 1. Spring Boot 配置

最小 standalone 配置：

```yaml
valkey:
  query:
    enabled: true
    mode: standalone
    nodes:
      - host: 127.0.0.1
        port: 6379
```

## 2. 读写分离

```yaml
valkey:
  query:
    mode: read_write_split
    read-preference: replica_preferred
    read-write:
      write:
        nodes:
          - host: 10.0.0.10
            port: 6379
      read:
        nodes:
          - host: 10.0.0.11
            port: 6380
```

说明：

- 写请求固定走主节点
- 只读请求按 `read-preference` 决定优先级
- 索引治理命令固定走写路径

## 3. Cluster

```yaml
valkey:
  query:
    mode: cluster
    cluster:
      nodes:
        - host: 10.0.1.10
          port: 6379
        - host: 10.0.1.11
          port: 6379
```

说明：

- 索引命令会广播到各分片
- 查询结果会做跨分片合并
- 只有文档明确承诺的聚合子集才支持 cluster
- cluster 具体能力边界直接看 [拓扑支持矩阵](/Users/momao/dm/java/demo/valkey-demo/docs/拓扑支持矩阵.md)

## 4. 索引治理

当前只保留 3 种模式：

- `NONE`
- `VALIDATE`
- `RECREATE`

### 4.1 `NONE`

- 什么都不做
- 启动最快
- 生产环境最稳

### 4.2 `VALIDATE`

- 只检查
- 一旦索引缺失或 schema 不一致，直接阻止启动

### 4.3 `RECREATE`

- 删除后重建
- 只适合本地开发和测试

## 5. 观测能力

当前支持：

- Observation
- 慢查询日志
- 失败指标
- 索引治理指标
- 局部更新指标

相关配置：

```yaml
valkey:
  query:
    observability:
      enabled: true
      slow-log-enabled: true
      async-slow-log-enabled: true
      slow-query-threshold-ms: 200
      trace-query-text-enabled: false
```

## 6. 当前治理边界

### 6.1 没有自动无损索引升级

当前没有基于 `FT.ALTER` 的自动升级能力。  
原因不是框架偷懒，而是当前服务端能力边界不稳定。

### 6.2 真实环境验证优先

这套框架现在明确遵守：

- 没有真实 Valkey 兜底的运行时能力，不算真正交付
- 本地已有索引和数据，不代表 CI fresh 容器也成立

## 7. 多模块 Spring Boot 接入

单模块项目推荐：

```java
@EnableValkeyQuery
```

多模块项目推荐：

```java
@EnableValkeyQuery({
    "com.example.account",
    "com.example.order",
    "com.example.search"
})
```

说明：

- 无参扫描只适合启动类和 Repository/实体在同一包树下的项目
- 只要进入多模块，优先显式写包
- 扫描不到时，先查 `@EnableValkeyQuery` 配置，而不是先怀疑 APT

## 8. 当前最重要的一个限制

`TAG` 字段当前不会生成 `SORTABLE`。

也就是说：

- `TEXT` / `NUMERIC` 可以排序
- `TAG` 不做 `SORTABLE`

这是按真实服务端能力收紧后的结果，不是文档保守写法。

## 9. 这轮真实验证补出来的两个索引边界

### 8.1 `TAG` 默认不生成 `SORTABLE`

这是当前框架已经落实到代码里的行为。

原因：

- 真实服务端下，`TAG + SORTABLE` 组合会导致 `FT.CREATE` 参数错位
- 最终表现就是后续字段名被当成前一个字段参数，直接建索引失败

### 8.2 `TEXT` 默认不生成 `WITHSUFFIXTRIE`

当前框架也不会再为普通 `TEXT` 字段默认追加 `WITHSUFFIXTRIE`。

原因：

- 注解层没有显式开关
- 真实服务端下，这个参数会把后续字段参数吃坏
- 框架不能靠“隐式增强”去赌服务端兼容性
